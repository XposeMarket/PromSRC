# Supervised coding and fresh-review modes

## Map before goal
Before implementation, split code, docs, runtime evidence, and unknowns into bounded mapping lanes. A planning owner attacks the premise and reconciles maps before locking the goal.

## Supervisor lane
Assign one owner to steering, scope, acceptance, conflict resolution, and the completion ledger. This lane must not absorb implementation by default.

## Fresh reviewer lane
The implementation owner writes a structured review brief with objective, criteria, changed artifacts, doubts, tests, and evidence paths. A reviewer with no implementation history inspects actual artifacts and adjudicates findings as confirmed, disproven, or unresolved. Use `independent-fresh-context-review` for the full protocol.

## Model-role matrix
Use measured routing rather than prestige: planning, coding, context/research, computer action, and review each receive a primary/fallback route supported by `evidence-driven-agent-model-router` evidence.

## Reasoning relevance check
For performance or architecture work, require profiling or path evidence that the suspected bottleneck is on the real execution/render path. Reject optimization of merely related components.

## Completion ledger
| Acceptance criterion | Artifact/evidence | Verifier | Status | Residual risk |
|---|---|---|---|---|

A lane reply is not completion. Reconcile reviewer objections and verify each criterion before merge or completion.
