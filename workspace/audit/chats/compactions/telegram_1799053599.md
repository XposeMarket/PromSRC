## The real bug

The bug is that `declare_plan` itself is being counted as a normal tool progress event.

After the custom `declare_plan` handler runs, it does this:

- seeds the declared plan
- sets skill-scout required
- then calls `continue`

But **before that**, the round already called:

- `markProgressStepStart(toolName)` at **4133**
- and for `declare_plan`, the progress state now thinks step 1 is already active

Then later, after `complete_plan_step`, the code does this:

- if skill scout is complete, it says:
  - `"Declared plan step 1 has NOT advanced; proceed with step 1 now."`
  - at **4300**
- but then it still calls:
  - `markProgressStepResult(true, toolName)` at **4319**

That means the progress tracker and the manual skill-scout gate are not perfectly aligned. The skill-scout pre-step is treated like a pseudo-step layered on top of step 1, but the progress machinery still treats tool calls as activity on step 1 itself.

### In plain English
The system has:
1. a hidden ‚Äúskill scout pre-step‚Äù
2. a visible declared step 1

But it does **not** cleanly separate those two states in progress tracking.

So you get this bad loop:
- `skill_list` ran
- `complete_plan_step` says success
- but next pass still thinks step 1 hasn‚Äôt properly satisfied the gate / cursor state
- then blocks again with ‚Äúrun skill_list first‚Äù

## My conclusion

You were right on the core issue:

- **It‚Äôs not that there is zero detection**
- it‚Äôs that the detection is **incomplete / miswired**
- specifically, the hidden skill-scout phase is not modeled as a first-class plan phase, so `complete_plan_step` and the progress cursor can disagree about whether step 1 is actually clear

## The likely fix direction

The clean fix is one of these:

1. **Best fix:** make skill scout a real explicit pre-step in plan state  
   - separate from declared step 1
   - no overloading of `stepCursor === 0`

2. **Simpler fix:** do not treat `declare_plan` / skill-scout tools as normal step progress events  
   - especially avoid progress mutations around `declare_plan`
   - and avoid `markProgressStepResult(true, toolName)` from muddying the step state after scout completion

3. **Even safer patch:** add a dedicated boolean like:
   - `manualPlanSkillScoutCompleted = true`
   - and gate `complete_plan_step` off that, instead of only `manualPlanSkillListDone` + cursor assumptions
