# Best execution model

You said:

> Every time the gateway opens it starts a 6 hr timer

That’s fine, but I’d refine it.

## Better model: heartbeat + state-based eligibility
Instead of a fragile literal 6-hour timer only, do this:

- on gateway start:
  - initialize brain runtime state
- run a lightweight periodic checker, e.g. every **15 minutes**
- checker decides:
  - has 6 hours elapsed since last thought?
  - is it nighttime and dream hasn’t run yet?
  - if yes, dispatch the relevant job

### Why this is better
A pure timer can be brittle if:
- restart happens,
- process sleeps,
- machine pauses,
- run crashes midway.

A checker + persisted state is much safer.

So internally the rule becomes:

- **Thought eligible** if `now - lastThoughtAt >= 6h`
- **Dream eligible** if local time is after configured nightly threshold and `lastDreamDate != today`

That gives you the same effect without depending on one in-memory timer.

# Risk areas to solve up front

## 1. Duplicate proposal spam
If Dream re-proposes the same thing every night, it’ll get annoying.

### Fix
Maintain dedupe logic:
- compare against existing pending proposals
- compare against yesterday’s proposal ledger
- compare title + affected files + semantic reason
- mark as:
  - submitted
  - already pending
  - deferred duplicate

This is important.

## 2. Memory pollution
If Dream writes too eagerly to `USER.md` / `SOUL.md`, those files become sludge.

### Fix
Memory write gate:
- must be durable
- must be useful in future sessions
- must not already exist
- must have evidence from repeated or strong signals

## 3. Weak proposals
If Dream creates vague proposals, the feature loses value.

### Fix
Proposal quality thresholds:
- no proposal without concrete target
- no src proposal without source inspection
- no schedule proposal without current schedule inspection
- no skill proposal without inspecting existing skill landscape first

## 4. Over-scanning cost
Scanning all files every 6 hours could get expensive/noisy.

### Fix
Use a windowed inventory first:
- identify files modified within window
- classify by type/path
- read selectively
- cap max volume
- prioritize:
  - audit
  - notes
  - proposals
  - tasks
  - memory files if touched
  - relevant configs
