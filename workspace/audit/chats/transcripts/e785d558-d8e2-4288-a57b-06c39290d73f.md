## My judgment on the core issue

I think your diagnosis is exactly right:

> the right way is for Prometheus to guide itself through workflows using proper screenshot/state-grounding so it can actually see what it’s doing

That should be the governing principle for both browser and desktop skills.

### The updated doctrine should be:

#### Browser
- start from **fresh `browser_open` / `browser_snapshot` evidence**
- after any action, use returned snapshot/delta instead of assuming result
- if DOM is sparse, misleading, or JS-heavy:
  - use `browser_vision_screenshot`
  - then `browser_vision_click` / `browser_vision_type`
- use `browser_run_js` only as fallback, not primary orientation
- use `browser_get_page_text` when text extraction is needed and snapshot is sparse
- use `browser_extract_structured` and network interception for data-heavy tasks
- trust **current visual state** over remembered refs or assumptions

#### Desktop
- start with **`desktop_screenshot` or `desktop_window_screenshot`**
- focus the target window before action
- after clicking/typing, re-screenshot or use change detection
- use `desktop_get_accessibility_tree` for exact control targeting
- use `desktop_get_window_text` when OCR is weak
- use `desktop_pixel_watch` / `desktop_wait_for_change` instead of blind sleeps
- avoid “guessing” coordinates when a screenshot/tree can resolve it

In other words:  
**observe → anchor → act → verify**  
not  
**assume → act → hope**
