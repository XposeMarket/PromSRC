# Report Generator

Lead with the outcome. Choose the smallest format that fits the destination: concise Markdown in chat, a saved Markdown artifact for handoff, HTML only when browser styling is required, or a document skill when the user requests a formal document.

## Report contract

1. Identify audience, decision, scope, time window, and source evidence.
2. Start with a 2–4 sentence plain-language summary.
3. Show exact metrics and denominators before interpretation.
4. Separate verified findings, inferences, warnings, and unknowns.
5. Put important failures near the top; never bury them after success language.
6. Cite sources or local artifacts where claims can be checked.
7. End with specific next actions, owners or dependencies when known, and blockers.
8. Save only when requested or required by the workflow, using a current workspace-relative destination rather than a hard-coded machine path.
9. Reopen the artifact and verify links, tables, encoding, and rendered layout when applicable.

Read [the detailed guide](references/detailed-guide.md) for Markdown/HTML templates and generation examples. Treat its historical output paths as examples, not current configuration.

Do not pad reports, invent precision, confuse absence of errors with proof of success, or expose secrets and private audit payloads.
